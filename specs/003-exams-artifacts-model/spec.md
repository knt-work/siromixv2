# Feature Specification: Extend Core Data Model with Exams and Artifacts

**Feature Branch**: `003-exams-artifacts-model`  
**Created**: March 12, 2026  
**Status**: Draft  
**Input**: User description: "Feature — Extend Core Data Model with Exams and Artifacts. Extend the current SiroMix database model by introducing new domain tables for exam records and generated artifacts, and connect them properly with the existing users and tasks tables."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - System stores exam business metadata independently from task execution (Priority: P1)

When a user creates a new exam through the Create New Exam form, the system must persist the exam's business-level information (name, subject, academic year, etc.) as a distinct record that exists independently of any processing tasks. This exam record serves as the parent entity for all related processing activities and generated outputs.

**Why this priority**: This is foundational - without a persistent exam entity, there is no way to organize tasks or artifacts around a business concept. This enables exam-level operations like viewing history, re-processing, and artifact management.

**Independent Test**: Can be fully tested by creating an exam record through the form submission, verifying it persists in the database with all required metadata fields, and confirming it can be retrieved independently of any task execution.

**Acceptance Scenarios**:

1. **Given** a user has submitted the Create New Exam form with complete metadata, **When** the system processes the submission, **Then** a new exam record is created with exam name, subject, academic year, grade level, number of variants, instructions, initial status, user ownership, and timestamps
2. **Given** an exam record exists in the database, **When** a user requests exam details, **Then** the system returns all business metadata without requiring any task to have been executed
3. **Given** multiple exams belong to the same user, **When** the user views their exam list, **Then** all exams are displayed with their business metadata regardless of processing state

---

### User Story 2 - Tasks are associated with their parent exam (Priority: P1)

Each task execution (pipeline processing job) must be linked to the exam it is processing. This enables tracking which tasks belong to which exam, supports re-running processing for the same exam, and ensures task execution history is organized by exam.

**Why this priority**: Critical for maintaining data integrity and enabling exam-level task management. Without this relationship, tasks become orphaned execution records with no business context.

**Independent Test**: Can be fully tested by creating a task for an exam, verifying the task record includes the exam foreign key, and confirming task queries can filter by exam.

**Acceptance Scenarios**:

1. **Given** an exam record exists, **When** a new task is created for that exam, **Then** the task record includes the exam ID as a foreign key
2. **Given** an exam has multiple tasks (initial run and retry), **When** querying tasks for that exam, **Then** all associated tasks are returned in chronological order
3. **Given** an exam is deleted, **When** the deletion cascades, **Then** all associated tasks are also removed from the database

---

### User Story 3 - System tracks generated artifacts linked to exams and tasks (Priority: P1)

When pipeline stages generate outputs (DIJ files, question previews, NES files, variants packages, answer matrices), the system must create artifact records that reference both the source exam and the producing task. This enables traceability from any artifact back to its business context and execution origin.

**Why this priority**: Essential for output management, debugging, and user access to results. Without artifact records, there is no way to know what files exist, where they came from, or how to serve them to users.

**Independent Test**: Can be fully tested by simulating artifact creation during task execution, verifying artifact records persist with exam ID, task ID, file metadata, and artifact type, and confirming artifacts can be queried by exam or by task.

**Acceptance Scenarios**:

1. **Given** a task is processing an exam through a pipeline stage, **When** the stage generates an output file, **Then** an artifact record is created with artifact type, exam reference, task reference, file name, file path, mime type, and timestamp
2. **Given** an exam has completed processing, **When** querying artifacts for that exam, **Then** all generated artifacts are returned including DIJ, question previews, NES, variants packages, and answer matrices
3. **Given** a task produced multiple artifacts across different stages, **When** querying artifacts by task ID, **Then** all artifacts generated by that specific task execution are returned
4. **Given** an exam is deleted, **When** the deletion cascades, **Then** all associated artifact records are removed (file cleanup is out of scope for this data model feature)

---

### User Story 4 - Exam ownership follows user identity (Priority: P2)

Exams must belong to individual users, ensuring proper access control foundations and supporting multi-tenant data isolation. Each exam record must reference the creating user.

**Why this priority**: Important for security and future access control, but can be implemented after basic CRUD operations work. The existing users table already supports this relationship pattern.

**Independent Test**: Can be fully tested by creating exams for different users, verifying each exam record includes the correct user ID, and confirming user-based exam queries return only owned exams.

**Acceptance Scenarios**:

1. **Given** a user is authenticated, **When** they create an exam, **Then** the exam record includes their user ID as a foreign key
2. **Given** multiple users exist in the system, **When** querying exams for a specific user, **Then** only exams owned by that user are returned
3. **Given** a user is deleted, **When** the deletion cascades, **Then** all exams owned by that user are removed along with their tasks and artifacts

---

### Edge Cases

- What happens when a task references a non-existent exam ID? (Database foreign key constraint prevents this)
- What happens when an artifact is created without a task reference? (Allowed - some artifacts may be manually uploaded or generated outside task execution)
- What happens when querying artifacts for an exam that has no completed tasks? (Returns empty list)
- What happens when multiple tasks for the same exam generate artifacts with the same type? (All artifacts persist - type alone does not enforce uniqueness)
- What happens when an exam is updated while tasks are running? (Exam metadata updates do not affect in-flight task execution - tasks operate on a snapshot)
- What happens when attempting to delete an exam with running tasks? (Cascade delete removes tasks regardless of status - worker must handle gracefully)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST provide an `exams` table to store exam business metadata independently from task execution state
- **FR-002**: System MUST provide an `artifacts` table to store references to generated outputs with exam and task associations
- **FR-003**: System MUST extend the `tasks` table with an `exam_id` foreign key to associate each task with its parent exam
- **FR-004**: System MUST maintain referential integrity through foreign key constraints for all relationships (user→exam, exam→task, exam→artifact, task→artifact)
- **FR-005**: System MUST support cascade deletion such that deleting a user removes all owned exams, their tasks, task logs, and artifacts
- **FR-006**: System MUST support cascade deletion such that deleting an exam removes all associated tasks, task logs, and artifacts
- **FR-007**: System MUST allow one user to own multiple exams (one-to-many)
- **FR-008**: System MUST allow one exam to have multiple tasks (one-to-many) to support retries and re-processing
- **FR-009**: System MUST allow one exam to have multiple artifacts (one-to-many)
- **FR-010**: System MUST allow one task to produce multiple artifacts (one-to-many)
- **FR-011**: System MUST require every exam to belong to exactly one user (mandatory foreign key)
- **FR-012**: System MUST require every task to belong to exactly one exam (mandatory foreign key)
- **FR-013**: System MUST require every artifact to belong to exactly one exam (mandatory foreign key)
- **FR-014**: System MUST allow artifacts to optionally reference the producing task (nullable foreign key)
- **FR-015**: System MUST store exam metadata including: exam name (VARCHAR 500), subject (VARCHAR 500), academic year (VARCHAR 50), grade/class level (VARCHAR 100, optional), number of variants (integer), instructions (TEXT, optional), exam-level status (enum: `draft`, `processing`, `completed`), and timestamps (created_at, updated_at)
- **FR-016**: System MUST store artifact metadata including: artifact type, file name, file path (VARCHAR 500, relative path from storage root following pattern `exams/<user_id>/<exam-name-kebab-case>/<filename>`), mime type, and timestamps (created_at)
- **FR-017**: System MUST support the following artifact types for MVP: `dij`, `question_preview`, `nes`, `variants_package`, `answer_matrix`
- **FR-018**: System MUST preserve the existing `users`, `tasks`, and `task_logs` table structures with only additive changes (adding `exam_id` to tasks)
- **FR-019**: System MUST ensure `exams` table stores only business metadata, not execution state
- **FR-020**: System MUST ensure `tasks` table stores only execution and processing state, not business metadata
- **FR-021**: System MUST ensure `artifacts` table stores only output references, not business or execution metadata
- **FR-022**: System MUST allow querying artifacts by exam ID to retrieve all outputs for a given exam
- **FR-023**: System MUST allow querying artifacts by task ID to retrieve all outputs from a specific task execution
- **FR-024**: System MUST allow querying tasks by exam ID to retrieve execution history for a given exam
- **FR-025**: System MUST allow querying exams by user ID to retrieve all exams owned by a user

### Key Entities

- **Exam**: Represents the top-level business entity for an exam document. Attributes include exam name (VARCHAR 500, required), subject (VARCHAR 500, required), academic year (VARCHAR 50, required), grade level (VARCHAR 100, optional), number of variants (required, integer), instructions (TEXT, optional), exam status (required enum: `draft`, `processing`, `completed`), user ID (required foreign key), created timestamp, updated timestamp. Each exam belongs to one user and can have many tasks and many artifacts.

- **Artifact**: Represents a generated output file or JSON result produced during task processing. Primary key is an auto-increment integer (serial/bigserial) for high-volume insert performance. Attributes include artifact type (required enum), exam ID (required foreign key), task ID (optional foreign key), file name (VARCHAR 255, required), file path (VARCHAR 500, required, relative path from storage root following pattern `exams/<user_id>/<exam-name-kebab-case>/<filename>`), mime type (VARCHAR 100, required), created timestamp. Each artifact belongs to one exam and optionally references one producing task.

- **Task** (extended): Existing async processing job entity, extended with exam ID (required foreign key). Each task now belongs to one exam. Existing attributes (status, stage, progress, logs) remain unchanged.

- **User** (unchanged): Existing user identity entity. Relationships extended to include one-to-many with exams.

- **TaskLog** (unchanged): Existing structured log entity. No changes to schema or relationships - continues to belong to tasks.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Database schema supports creating exam records with all required metadata fields and retrieving them independently from any task execution
- **SC-002**: Database schema enforces referential integrity such that every task must reference a valid exam, and every exam must reference a valid user
- **SC-003**: Database schema supports querying all tasks for a given exam and returns results in chronological order
- **SC-004**: Database schema supports querying all artifacts for a given exam and groups them by type (DIJ, NES, etc.)
- **SC-005**: Database schema supports cascade deletion such that deleting a user removes all owned exams and their associated tasks, logs, and artifacts with zero orphaned records
- **SC-006**: Database schema supports storing metadata for all five MVP artifact types (DIJ, question preview, NES, variants package, answer matrix)
- **SC-007**: Database schema allows creating multiple tasks for the same exam to support retry and re-processing scenarios
- **SC-008**: Database migration from current schema (users, tasks, task_logs) to extended schema (adding exams, artifacts, exam_id to tasks) completes without data loss
- **SC-009**: Database schema supports over 10,000 exam records per user without performance degradation on standard queries
- **SC-010**: Database schema maintains clear separation of concerns with business metadata in exams, execution state in tasks, and output references in artifacts

## Assumptions

- The existing `users`, `tasks`, and `task_logs` tables are already deployed and may contain production data
- Migration will add the `exams` table and `artifacts` table as new entities
- Migration will add an `exam_id` column to the existing `tasks` table as a required foreign key
- For any existing tasks in the database before migration, a default "Legacy Import" exam will be created per user with status `completed`, and all orphaned tasks will be linked to their user's legacy exam to maintain referential integrity
- File storage implementation (where artifact files actually live) is out of scope - this feature only tracks metadata references
- Artifact cleanup (deleting actual files when artifact records are removed) is out of scope - this feature only handles database records
- The Create New Exam form UI already exists or is being developed in parallel - this feature only provides the persistence layer
- The five MVP artifact types (DIJ, question preview, NES, variants package, answer matrix) are sufficient for initial release
- Artifact versioning (tracking multiple versions of the same artifact type for an exam) is out of scope for MVP

## Clarifications

### Session 2026-03-12

- Q: Artifact table primary key type - should it use auto-increment integer (like TaskLog for performance), UUID (consistent with business entities), or composite key? → A: Auto-increment integer (serial/bigserial) - optimized for performance like task_logs
- Q: Exam status field values - what are the valid status values for the exam-level status field? → A: Simple 3-state: `draft`, `processing`, `completed`
- Q: Field length constraints for text fields - what are the VARCHAR length limits for exam text fields (name, subject, academic year, grade level, instructions)? → A: Generous: name/subject=500, year=50, grade=100, instructions=TEXT
- Q: Artifact file path storage format - should paths be relative, full URLs, storage keys only, or both? → A: Relative path from storage root following the pattern `exams/<user_id>/<exam-name-kebab-case>/<filename>` (e.g., `exams/abc-123/mathematics-final-2026/dij-final.json`)
- Q: Migration strategy for existing tasks - how should pre-migration tasks without an exam be handled when exam_id becomes required? → A: Create a default "Legacy Import" exam per user to adopt orphaned tasks

- Implementation of exam review or artifact download UI
- Defining task status values or stage enumeration changes
- Defining task log structure changes
- Google OAuth integration or authentication logic
- Authorization rules for multi-user access control (schema supports it but enforcement is separate)
- Artifact versioning or history tracking
- Artifact file cleanup on deletion
- Performance optimization or indexing strategy (covered in implementation planning)
- Database migration rollback strategy
- Handling of existing tasks that predate the exams table
